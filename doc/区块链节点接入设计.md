# 区块链节点接入设计

## 一、What：什么是区块链节点

### 1.1 节点的本质

区块链节点是连接到区块链网络的计算机程序，负责：
- 存储区块链数据（区块、交易、状态）
- 验证交易和区块的有效性
- 广播交易到网络
- 与其他节点同步状态

```
┌─────────────────────────────────────────────────────────────┐
│                    区块链 P2P 网络                          │
│                                                              │
│    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐ │
│    │  节点A  │◄──►│  节点B  │◄──►│  节点C  │◄──►│  节点D  │ │
│    │(全节点) │    │(全节点) │    │(轻节点) │    │(全节点) │ │
│    └─────────┘    └─────────┘    └─────────┘    └─────────┘ │
│         │              │              │              │       │
│         └──────────────┴──────────────┴──────────────┘       │
│                      P2P 通信 + 状态同步                     │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 节点类型

| 类型 | 存储内容 | 功能 | 资源需求 |
|------|---------|------|---------|
| **全节点 (Full Node)** | 完整区块链数据 | 验证所有交易、参与共识 | 高（TB级存储） |
| **归档节点 (Archive Node)** | 完整数据 + 历史状态 | 查询任意历史状态 | 极高 |
| **轻节点 (Light Node)** | 仅区块头 | 验证交易存在性 | 低 |
| **验证节点 (Validator)** | 完整数据 | 产生区块、参与共识 | 高 + 质押 |

### 1.3 节点与钱包系统的关系

```
钱包系统 ──RPC调用──▶ 节点 ──P2P网络──▶ 区块链网络
              (JSON-RPC)    (gossip协议)
```

**节点为钱包系统提供的能力：**
- 订阅新区块/新交易
- 查询地址余额
- 查询交易详情
- 广播已签名的交易
- 估算 Gas 费用

## 二、Why：为什么节点接入是关键设计

### 2.1 可靠性依赖

钱包系统的核心能力完全依赖节点：
- **充值监听**：节点必须稳定推送新区块事件
- **提现广播**：节点必须成功将交易广播到网络
- **余额查询**：节点数据必须准确且及时

**节点故障的后果：**
```
节点不可用 → 充值事件丢失 → 用户资金未入账 → 用户投诉
节点延迟高 → 提现确认慢 → 用户体验差
节点数据错 → 余额显示错误 → 资金风险
```

### 2.2 成本与性能的权衡

| 方案 | 成本 | 性能 | 可控性 | 风险 |
|------|------|------|-------|------|
| 自建全节点 | 高（硬件+带宽） | 高 | 完全可控 | 维护负担 |
| 第三方 RPC | 中（按调用付费） | 中高 | 依赖服务商 | 服务商限流/故障 |
| 公共节点 | 免费 | 低 | 无 | 不稳定、随时失效 |

### 2.3 去中心化与中心化的矛盾

**理想状态：** 每个钱包系统运行自己的全节点，最大化去中心化
**现实约束：**
- 全节点存储成本高（ETH 全节点 > 1TB）
- 同步时间长（ETH 同步需数天）
- 维护需要专业团队

**现实平衡：** 使用第三方 RPC 服务（引入中心化依赖）

## 三、How：如何接入节点

### 3.1 三种接入方案

#### 方案一：自建全节点

```
硬件准备 → 下载客户端 → 同步区块链 → 开启 RPC → 接入服务
```

**各链自建成本：**

| 公链 | 存储需求 | RAM | 同步时间 | 带宽 |
|------|---------|-----|---------|------|
| BTC | ~500GB | 2GB+ | 数天 | 中 |
| ETH | ~1.5TB（快照模式） | 16GB+ | 数天 | 高 |
| TRX | ~数TB | 32GB+ | 数天 | 高 |
| SOL | ~数TB | 32GB+ | 数天 | 极高 |

**适用场景：** 大型交易所、对数据主权要求高的机构

#### 方案二：第三方 RPC 服务

```
注册服务商 → 获取 API Key → 配置 Endpoint → 开始调用
```

**主流服务商：**

| 服务商 | 支持链 | 特点 |
|--------|-------|------|
| Infura | ETH、BTC、Polygon | 以太坊首选，稳定性高 |
| Alchemy | ETH、ARB、OP、MATIC | 开发友好，有免费额度 |
| QuickNode | ETH、SOL、BTC | 多链支持，性能好 |
| Blockstream | BTC | 比特币专业服务 |
| Ankr | 多链 | 免费公共节点 |

**适用场景：** 中小型项目、快速开发、降低运维成本

#### 方案三：公共节点

```
直接使用公开的 RPC Endpoint → 无需注册 → 调用受限
```

**示例：**
- ETH: `https://eth.llamarpc.com`
- BTC: `https://blockstream.info/api`
- SOL: `https://api.mainnet-beta.solana.com`

**风险：**
- 可能随时失效
- 有速率限制
- 不保证可用性

**适用场景：** 开发测试、原型验证

### 3.2 生产环境推荐架构

```
┌─────────────────────────────────────────────────────────────┐
│                    钱包服务                                  │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐  │
│  │              节点发现与故障切换                     │  │
│  └──────────────────────────────────────────────────────┘  │
│                           │                                  │
│         ┌─────────────────┼─────────────────┐               │
│         ▼                 ▼                 ▼               │
│  ┌─────────┐        ┌─────────┐        ┌─────────┐        │
│  │主节点组  │        │备用节点组│        │公共节点 │        │
│  │(付费RPC)│        │(付费RPC)│        │(降级使用)│        │
│  └─────────┘        └─────────┘        └─────────┘        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**故障切换策略：**
1. 主节点组故障 → 切换到备用节点组
2. 备用节点组故障 → 切换到公共节点（降级服务）
3. 定期健康检查与告警

## 四、节点接入的关键设计

### 4.1 调用限流与重试

```
RPC 调用 ──▶ 限流器 ──▶ 节点
              │
      超限 ──┤
              ▼
         等待/重试
```

**策略：**
- 令牌桶限流（避免触发服务商封禁）
- 指数退避重试（避免雪崩）
- 熔断机制（连续失败后暂停）

### 4.2 数据一致性保障

**问题：** 不同节点返回的数据可能有短暂差异

**解决方案：**
- 关键操作（余额查询）调用多个节点验证
- 使用确认数策略（等待 N 个区块确认）
- 定期对账与节点数据校验

### 4.3 安全考虑

- RPC Endpoint 加密（HTTPS）
- API Key 密钥管理（不进代码仓库）
- 节点响应白名单（避免钓鱼攻击）

---

## 五、总结

区块链节点是钱包系统连接链上世界的唯一通道。自建节点可控但成本高，第三方 RPC 便捷但有依赖风险。生产环境推荐采用主备节点组 + 故障切换的架构，在不同服务商之间分散风险，保障服务的高可用性。
