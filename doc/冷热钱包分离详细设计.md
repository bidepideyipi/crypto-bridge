# 冷热钱包分离详细设计

## 版本说明

**当前版本：** v0.1.0 开发版

---

## 一、设计目标

### 1.1 核心目标

| 目标 | 说明 |
|------|------|
| **资金安全** | 大额资金存储在冷钱包，私钥永不触网 |
| **自动化** | 热钱包自动归档，减少人工操作 |
| **权限分离** | 多签审批，防止单点故障 |
| **可审计** | 所有操作留痕，支持审计追踪 |

### 1.2 冷热钱包定位

```
┌─────────────────────────────────────────────────────────────┐
│                      热钱包 (Hot Wallet)                     │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  用途：日常小额提现                                   │  │
│  │  容量：保持最小运营余额                               │  │
│  │  签名：自动签名（HSM）                                 │  │
│  │  归档：超额自动归档到冷钱包                            │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                              │ 归档
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      冷钱包 (Cold Wallet)                     │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  用途：大额资金存储                                   │  │
│  │  容量：大部分资金                                       │  │
│  │  签名：多签审批（人工）                                 │  │
│  │  触发：大额提现需先转入热钱包                            │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

## 三、Shamir 秘钥分片方案

### 3.1 算法原理

Shamir's Secret Sharing (SSS) 将私钥拆分成 N 份，任意 K 份可以重组私钥。

```
原始私钥 (32 bytes)
      │
      ▼
┌─────────────────────────────────────────────────────────────┐
│                  Shamir 秘钥分片                              │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  配置：(3, 5) - 任意 3 份可重组                       │  │
│  │                                                        │  │
│  │  Share 1: 持有人 A (财务总监)                        │  │
│  │  Share 2: 持有人 B (财务经理)                        │  │
│  │  Share 3: 持有人 C (CEO)                            │  │
│  │  Share 4: 持有人 D (CTO)                             │  │
│  │  Share 5: 持有人 E (安全总监)                        │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
      │
      ▼ 任意 3 人共同在场
┌─────────────────────────────────────────────────────────────┐
│                   秘钥重组                                    │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  Share 1 + Share 2 + Share 3 → 原始私钥              │  │
│  │  Share 1 + Share 3 + Share 5 → 原始私钥              │  │
│  │  Share 2 + Share 4 + Share 5 → 原始私钥              │  │
│  │  ... (任意组合)                                       │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 Go 实现

```go
package shamir

import (
    "crypto/rand"
    "github.com/hashicorp/vault/shamir"
)

// Split 秘钥分片
func Split(secret []byte, n, threshold int) ([][]byte, error) {
    return shamir.Split(secret, n, threshold)
}

// Combine 秘钥重组
func Combine(shares [][]byte) ([]byte, error) {
    return shamir.Combine(shares)
}

// GenerateColdWalletKey 生成冷钱包密钥并分片
func GenerateColdWalletKey(n, threshold int) ([][]byte, error) {
    // 1. 生成 32 字节随机私钥
    secret := make([]byte, 32)
    if _, err := rand.Read(secret); err != nil {
        return nil, err
    }

    // 2. 使用 Shamir 算法分片
    shares, err := shamir.Split(secret, n, threshold)
    if err != nil {
        return nil, err
    }

    // 3. 安全删除原始私钥（只保留分片）
    // ...

    return shares, nil
}
```

### 4.2 多签地址生成

#### Bitcoin (P2SH)

```go
package btc

import (
    "github.com/btcsuite/btcd/btcec/v2"
    "github.com/btcsuite/btcd/btcutil"
    "github.com/btcsuite/btcd/chaincfg"
    "github.com/btcsuite/btcd/txscript"
)

// CreateMultisigAddress 创建 Bitcoin 多签地址
func CreateMultisigAddress(pubKeys []*btcec.PublicKey, m, n int) (string, error) {
    // 1. 构建多签赎回脚本
    builder := txscript.NewScriptBuilder()
    builder.AddInt64(int64(m))
    for _, pubkey := range pubKeys {
        builder.AddData(pubkey.SerializeCompressed())
    }
    builder.AddInt64(int64(n))
    builder.AddOp(txscript.OP_CHECKMULTISIG)

    script, err := builder.Script()
    if err != nil {
        return "", err
    }

    // 2. 生成 P2SH 地址
    address, err := btcutil.NewAddressScriptHash(
        script,
        &chaincfg.MainNetParams,
    )
    if err != nil {
        return "", err
    }

    return address.String(), nil
}
```

#### Ethereum (多签合约)

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract MultiSigWallet {
    uint256 public required;
    mapping(address => bool) public isOwner;
    address[] public owners;
    uint256 public transactionCount;

    struct Transaction {
        address destination;
        uint256 value;
        bytes data;
        bool executed;
    }

    mapping(uint256 => Transaction) public transactions;
    mapping(uint256 => mapping(address => bool)) public confirmed;

    constructor(address[] memory _owners, uint256 _required) {
        require(_owners.length >= _required);
        require(_required >= 1);
        for (uint256 i = 0; i < _owners.length; i++) {
            isOwner[_owners[i]] = true;
            owners.push(_owners[i]);
        }
        required = _required;
    }

    function confirmTransaction(uint256 _txIndex) public {
        require(isOwner[msg.sender]);
        Transaction storage tx = transactions[_txIndex];
        require(!tx.executed);

        confirmed[_txIndex][msg.sender] = true;

        uint256 count = 0;
        for (uint256 i = 0; i < owners.length; i++) {
            if (confirmed[_txIndex][owners[i]]) {
                count += 1;
            }
        }

        if (count == required) {
            executeTransaction(_txIndex);
        }
    }
}
```

│  │  - 切换备用节点                                         │  │
│  │  - 重试广播                                             │  │
│  │                                                        │  │
│  │  场景3：交易被拒绝                                       │  │
│  │  - 检查手续费是否足够                                    │  │
│  │  - 检查余额是否变化                                     │  │
│  │  - 重新构建交易                                         │  │
│  │                                                        │  │
│  │  场景4：长时间未确认                                     │  │
│  │  - 使用 RBF (Replace-By-Fee) 加费                      │  │
│  │  - 或取消交易，重试归档                                  │  │
│  └─────────────────────────────────────────────────────┘  │

## 六、冷热钱包切换

### 6.1 提现金额分级

```
┌─────────────────────────────────────────────────────────────┐
│                提现金额分级策略                              │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  小额提现 (< 热钱包余额)                               │  │
│  │  → 直接从热钱包支付                                     │  │
│  │  → HSM 自动签名                                        │  │
│  │  → 立即广播                                            │  │
│  └─────────────────────────────────────────────────────┘  │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  大额提现 (> 热钱包余额)                               │  │
│  │  → 先从冷钱包转出到热钱包                               │  │
│  │  → 触发多签审批                                        │  │
│  │  → 审批通过后转出                                      │  │
│  │  → 再从热钱包支付给用户                                 │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 大额提现完整流程

```
用户发起大额提现
       │
       ▼
┌─────────────────────────────────────────────────────────────┐
│              1. 检查热钱包余额                                │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  hot_balance = 查询热钱包余额()                        │  │
│  │  if hot_balance < 提现金额:                             │  │

│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────────────────────────────┐
│              2. 创建冷钱包转出审批                            │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  amount = 提现金额 + 手续费 + 缓冲                     │  │
│  │  创建多签审批                                          │  │
│  │  related_type = "cold_to_hot"                          │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────────────────────────────┐

│  ┌─────────────────────────────────────────────────────┐  │

│  │  - 重组冷钱包私钥签名                                  │  │

└─────────────────────────────────────────────────────────────┘
       │
       ▼

│  │  - 热钱包余额增加                                      │  │
│  │  - 冷钱包余额减少                                      │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────────────────────────────┐
│              5. 执行用户提现                                  │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  - 从热钱包构建提现交易                                │  │
│  │  - HSM 自动签名                                        │  │
│  │  - 广播到链上                                          │  │
│  │  - 完成提现                                            │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘

### 7.2 操作审计

```
所有冷钱包相关操作必须记录审计日志：

| 操作类型 | 记录内容 |
|---------|---------|
| 归档触发 | 触发原因、金额、操作人 |
| HSM 签名 | Key ID、交易哈希、时间戳 |
| 多签创建 | 审批 ID、关联业务、金额 |
| 多签签名 | 签名者、签名时间、IP 地址 |
| 冷钱包转出 | 转出金额、目标地址、审批者 |
| 密钥轮换 | 旧密钥 ID、新密钥 ID、操作人 |
| 分片重组 | 参与者、时间、用途 |
```

```
┌─────────────────────────────────────────────────────────────┐
