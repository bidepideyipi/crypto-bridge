# 冷钱包服务设计

## 一、What：什么是冷钱包服务

### 1.1 冷钱包的本质

冷钱包不是"购买的服务"，而是一种**安全隔离架构**。核心思想是：私钥永远不接触联网设备。

```
┌─────────────────────────────────────────────────────────────┐
│                      热钱包（在线）                          │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  私钥在线 │ 自动签名 │ 小额支付 │ 日常运营             │  │
│  │  (有风险)   │          │          │                  │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                              │
                        资金归档
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      冷钱包（离线）                          │
│  ┌─────────────────────────────────────────────────────┐  │
│  │  私钥离线 │ 人工审批 │ 大额存储 │ 资金库               │  │
│  │  (高安全)   │ 多签验证 │          │                  │  │
│  └─────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 冷钱包的实现形式

| 形式 | 说明 | 成本 | 安全级别 |
|------|------|------|---------|
| **硬件钱包** | Ledger、Trezor 等设备 | 数百-数千元 | 高 |
| **HSM（硬件安全模块）** | 专业加密设备 | 数万-数十万 | 极高 |
| **气隙隔离** | 完全物理隔离的离线电脑 | 低（已有设备） | 极高 |
| **Shamir 秘钥分片** | 多人持有多份密钥碎片 | 0（方案设计） | 极高 |
| **云 HSM 服务** | AWS CloudHSM、Azure HSM | 按小时付费 | 高 |

## 二、Why：为什么需要冷钱包

### 2.1 安全新手常见误区

```
❌ "冷钱包是购买的服务"
   → 冷钱包是架构设计，不是购买的服务

❌ "代码加密就安全了"
   → 私钥在内存中仍可被提取

❌ "数据库加密就安全了"
   → 有数据库访问权限就能拿到私钥
```

### 2.2 热钱包的风险场景

```
服务器被入侵 → 私钥被窃取 → 资金归零
内部人员作恶 → 导出私钥 → 资金被盗
代码漏洞泄露 → 私钥暴露 → 资金损失
```

### 2.3 冷热分离的核心价值

| 维度 | 热钱包 | 冷钱包 |
|------|-------|-------|
| 私钥存储 | 在线服务器 | 离线设备/硬件 |
| 签名方式 | 自动签名 | 人工/多签审批 |
| 资金规模 | 小额（日常运营） | 大额（资金库） |
| 风险暴露 | 高 | 低 |
| 用户体验 | 快 | 慢（需审批） |

**策略：** 热钱包保持最小余额，超额资金自动归档到冷钱包

## 三、How：冷钱包如何实现

### 3.1 生产环境方案

#### 方案一：硬件钱包（推荐中小型项目）

```
设备：Ledger Nano X / Trezor Model T
连接：USB / 蓝牙（仅传输待签名数据）
签名：设备物理按键确认
```

**优势：**
- 成本低（~1000 元）
- 即插即用
- 私钥永不离设备

**适用场景：** 中小型交易所、钱包项目

#### 方案二：HSM 硬件安全模块（推荐大型项目）

```
硬件：专业 HSM 设备（如 Thales、SafeNet）
或云服务：AWS CloudHSM、Azure Dedicated HSM
特性：FIPS 140-2 Level 3 认证
```

**优势：**
- 金融级安全
- 防物理篡改
- 可审计日志

**适用场景：** 大型交易所、金融机构

#### 方案三：Shamir 秘钥分片（推荐去中心化团队）

```
原理：将私钥拆分成 N 份，需要 K 份才能重组
示例：5 人各持 1 份，需要 3 人同时在场才能签名
```

**优势：**
- 无单点故障
- 防止单人作恶
- 零硬件成本

**适用场景：** 分布式团队、DAO

**实现库：**
- Go: `github.com/hashicorp/vault/shamir`

#### 方案四：气隙隔离（推荐极简方案）

```
离线电脑（无网卡） + 纸质备份
```

**流程：**
```
在线电脑生成未签名交易 → U盘 → 离线电脑签名 → U盘 → 在线电脑广播
```

### 3.2 开发环境模拟

#### 模拟方案一：纯软件模拟

```go
// 不使用真实私钥，使用测试链
type ColdWalletMock struct {
    privateKey string // 测试私钥，非真实资金
}

func (w *ColdWalletMock) Sign(tx *Transaction) (*Signature, error) {
    // 模拟签名流程，但使用的是测试私钥
    return w.privateKey.Sign(tx)
}
```

**注意：** 开发环境使用测试网（Testnet），私钥无真实价值

#### 模拟方案二：Docker 隔离

```bash
# 创建完全隔离的容器模拟冷钱包环境
docker run --network none --name cold-wallet-mock \
    -v $(pwd)/mock-key:/key:ro \
    alpine sh -c "read key; sign"
```

#### 模拟方案三：多签模拟

```
开发环境模拟 3-of-5 多签：
- 使用测试链地址
- 生成 5 个测试私钥
- 任意 3 个即可完成签名
- 验证多签逻辑
```

#### 模拟方案四：集成开发钱包

```
使用 MetaMask / Phantom 等开发钱包：
- 浏览器插件充当"冷钱包"
- 本地私钥（测试链）
- 模拟手动审批流程
```

### 3.3 各公链冷钱包实现

#### Bitcoin (BTC)

| 方案 | 工具/服务 |
|------|----------|
| 硬件钱包 | Ledger、Trezor |
| 多签 | Bitcoin Core 2-of-3 |
| 气隙 | Electrum（离线模式） |

#### Ethereum (ETH)

| 方案 | 工具/服务 |
|------|----------|

| 方案 | 工具/服务 |
|------|----------|
| 硬件钱包 | Ledger（Solana App） |

| 离线签名 | Solana CLI（离线模式） |

## 四、冷钱包操作流程

### 4.1 资金归档（热→冷）

```
┌─────────────────────────────────────────────────────────────┐
│  1. 监控热钱包余额                                          │
│      热钱包余额 > 阈值（如 10 BTC）？                      │
│                                                           │
│  2. 构造归档交易                                            │
│      From: 热钱包地址                                      │
│      To: 冷钱包地址                                        │
│      Amount: 超额部分                                      │
│                                                           │
│  3. 热钱包签名（自动）                                      │
│      HSM/自动签名服务                                      │
│                                                           │
│  4. 广播到链上                                              │
│      通过 RPC 节点广播                                    │
│                                                           │
│  5. 等待确认                                                │
│      等待安全确认数                                        │
│                                                           │
│  6. 更新内部账本                                            │
│      热钱包余额 -，冷钱包余额 +                           │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 大额提现（冷→热）

```
┌─────────────────────────────────────────────────────────────┐
│  1. 用户提现请求                                            │
│      用户发起大额提现（如 > 1 BTC）                         │
│                                                           │
│  2. 风控审核                                                │
│      风控系统检查（AML、KYC、行为分析）                     │
│                                                           │
│  3. 审批触发                                                │
│      创建多签审批任务                                      │
│                                                           │
│  4. 冷钱包签名（人工/多签）                                 │
│      ┌─────────────────────────────────────────────────┐  │
│      │  气隙方案：离线电脑签名                           │  │
│      │  硬件钱包：物理按键确认                           │  │
│      │  多签方案：K-of-N 人签名                          │  │
│      └─────────────────────────────────────────────────┘  │
│                                                           │
│  5. 广播到链上                                              │
│      通过 RPC 节点广播                                    │
│                                                           │
│  6. 监听确认                                                │
│      等待链上确认                                          │
│                                                           │
│  7. 完成提现                                                │
│      通知用户、更新账本                                    │
└─────────────────────────────────────────────────────────────┘
```

## 五、常见误区与安全建议

### 5.1 常见误区

```
❌ "开发环境不需要冷钱包"
   → 应该模拟完整流程，测试冷热切换

❌ "冷钱包私钥备份在云盘"
   → 失去冷钱包的意义

❌ "冷钱包私钥只有一份"
   → 丢失 = 资金归零

❌ "多人知道冷钱包私钥"
   → 增加泄露风险
```

### 5.2 安全建议

| 建议项 | 说明 |
|--------|------|
| **多重备份** | 冷钱包私钥至少 3 份异地备份 |
| **定期演练** | 每季度演练冷钱包恢复流程 |
| **权限分离** | 知道私钥的人 ≠ 能操作服务器的人 |
| **审计日志** | 所有冷钱包操作必须有日志 |
| **渐进启用** | 先小资金测试，再逐步增加额度 |

---

## 六、总结

冷钱包不是购买的服务，而是一种安全架构设计。中小型项目可采用硬件钱包（~1000 元），大型项目可考虑 HSM 或云 HSM 服务。开发环境应使用测试网模拟完整流程。冷热分离是保障大额资金安全的核心策略。
