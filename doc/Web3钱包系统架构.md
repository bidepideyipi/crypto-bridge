## 一、系统定位

多链钱包系统是连接区块链网络与中心化交易所（CEX）的核心桥梁，负责：
- 监听多公链的链上交易事件
- 维护钱包地址与用户账户的映射关系
- 实现冷热钱包分离的资金安全机制
- 保障链上账本与交易所账本的最终一致性

**关注点分离：**
本系统专注于钱包核心功能，通过定义清晰的服务接口与交易所业务层解耦。开发版本通过测试用例模拟业务层请求，验证服务层功能的正确性。

**接口定义：** 详见 `钱包服务接口定义.md`

## 三、核心组件设计

### 3.1 链适配层（Go实现）

每个公链对应一个独立的Adapter，封装链特异性：

```go
// 统一的链接口
type ChainAdapter interface {
    // 订阅链上事件
    SubscribeTransactions(ctx context.Context, addr Address) <-chan Transaction

    // 生成新地址
    GenerateAddress(userID string) (Address, error)

    // 广播交易
    BroadcastTx(tx *Transaction) (*TxReceipt, error)

    // 获取余额
    GetBalance(addr Address) (Balance, error)

    // 估算Gas
    EstimateGas(tx *Transaction) (uint64, error)
}

// BTC实现
type BitcoinAdapter struct {
    client *rpc.Client
    chain  ChainParams
}

// ETH实现
type EthereumAdapter struct {
    client *ethclient.Client
    chain  *params.ChainConfig
}
```

**技术选型：**
- BTC: `btcd` (Go官方实现)
- ETH: `go-ethereum` (Geth)
- TRX: ⏸️ 后续版本
- SOL: ⏸️ 后续版本

### 3.2 Goroutine-per-Chain并发模型

```go
// 每条链独立的监听Goroutine
func (s *Service) StartChainListeners(ctx context.Context) {
    for _, chain := range s.chains {
        go func(c ChainAdapter) {
            for {
                select {
                case tx := <-c.SubscribeTransactions(ctx):
                    s.processTransaction(tx)
                case <-ctx.Done():
                    return
                }
            }
        }(chain)
    }
}
```

**优势：**
- 单链故障不影响其他链
- 独立的错误重试机制
- 可按链特性优化监听策略

### 3.5 冷热钱包分离机制

#### 热钱包（Hot Wallet）
- **用途**：日常小额提现（< 阈值A）
- **特征**：在线、自动签名、低余额
- **安全**：HSM模块保护私钥、每日自动归档

#### 冷钱包（Cold Wallet）
- **用途**：大额资金存储、大额提现（≥ 阈值B）
- **特征**：离线、多人审批、多签机制
- **安全**：
  - Shamir Secret Sharing（门限签名）
  - 硬件钱包（Trezor/Ledger）
  - 物理隔离

#### 自动归档策略
```
热钱包余额 > 阈值 → 触发归档流程
                         │
                         ▼
                  生成归档交易（热→冷）
                         │
                         ▼
                  多签审批（K-of-N）
                         │
                         ▼
                  广播到链上确认
                         │
                         ▼
                  更新内部账本
```

## 四、账本一致性保障

### 4.1 最终一致性模型

```
链上状态 ←───────────────┐
    │                     │
    │ 实时监听            │ 定期对账
    ▼                     │
钱包服务账本              │
    │                     │
    │ 事件推送            │
    ▼                     │
消息队列 ◄────────────────┘
    │
    ▼
交易所账本服务
    │
    ▼
用户余额
```

### 4.2 对账机制

**实时对账：**
- 每个充值事件生成唯一追踪ID
- MQ确保At-Least-Once投递
- 消费端幂等处理

**定时对账：**
```
每日凌晨 → 从链上全量扫描钱包地址交易
              │
              ▼
         与数据库账本比对
              │
              ▼
      ┌───────┴───────┐
      │               │
      ▼               ▼
   差异上报      自动修复
```

## 八、故障恢复策略

**链节点故障：**
- 自动切换备用节点
- RPC节点轮询

**消息队列堆积：**
- 消费者水平扩展
- 优先级队列（充值优先于提现）

**数据库主从切换：**
- 读写分离
- 故障自动转移

## 一句话总结

> **多链钱包系统采用Go语言构建，通过Goroutine-per-Chain并发模型监听BTC/ETH公链（当前版本），使用Channel将链上事件桥接到消息队列，实现与交易所账本的最终一致性；基于HSM/Shamir Secret Sharing实现冷热钱包分离，通过K-of-N多签机制保障大额资金安全。开发版本使用公共节点，后续版本将支持更多公链及第三方渠道对接。**
