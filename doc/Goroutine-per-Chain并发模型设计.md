# Goroutine-per-Chain 并发模型设计

## 一、理论基础：Goroutine vs 线程

### 1.1 Goroutine 设计思想解决了什么问题

**传统线程模型的困境：**

| 问题 | 传统线程（OS线程） | 影响 |
|------|-------------------|------|
| 资源消耗 | 栈内存 ~1-2MB | 无法创建大量线程 |
| 调度开销 | 用户态 ↔ 内核态切换 | 频繁切换性能损失大 |
| 创建成本 | 需要 syscalls 调用 | 启动慢 |
| 数量限制 | 几千个已是上限 | 难以应对高并发场景 |

**Goroutine 的解决方案：**

```
┌─────────────────────────────────────────────────────────────┐
│                    Go Runtime (用户态)                      │
│  ┌───────────────┐    ┌───────────────┐                    │
│  │   Goroutine   │    │   Goroutine   │    ... 10万+      │
│  │   (2KB 栈)    │    │   (2KB 栈)    │                    │
│  └───────────────┘    └───────────────┘                    │
│           │                     │                          │
│           └──────────┬──────────┘                          │
│                      ▼                                     │
│              ┌───────────────┐                             │
│              │ M:N 调度器     │                             │
│              │ (GMP 模型)    │                             │
│              └───────────────┘                             │
│                      │                                     │
│         ┌────────────┼────────────┐                       │
│         ▼            ▼            ▼                       │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐                │
│   │ OS 线程1 │  │ OS 线程2 │  │ OS 线程N │   (通常=CPU数)  │
│   └──────────┘  └──────────┘  └──────────┘                │
└─────────────────────────────────────────────────────────────┘
```

**核心特性：**
- **轻量级**：初始栈仅 2KB，按需动态扩缩容
- **M:N 调度**：M 个 Goroutine 映射到 N 个 OS 线程
- **用户态调度**：无需陷入内核，切换快
- **数量无限制**：可轻松创建 10 万+ Goroutine

### 1.2 Goroutine vs Java 线程：根本差异

| 维度 | Goroutine (Go) | Java Thread | 差异本质 |
|------|----------------|-------------|---------|
| 模型 | 用户态轻量级线程 | 内核级重量级线程 | 调度层级不同 |
| 栈大小 | 2KB 起步，动态扩容 | ~1MB 固定 | 内存效率差 500 倍 |
| 创建速度 | 微秒级 | 毫秒级 | 快 1000 倍+ |
| 切换开销 | 纯用户态 | 用户态+内核态 | 少一次系统调用 |
| 典型数量 | 10 万+ | 几千 | 数量级差异 |
| 调度器 | Go Runtime (GMP) | OS Kernel | 谁来调度 |
| 阻塞处理 | 优雅 (让出 CPU) | 占用线程等待 | 资源利用率 |

**为什么 Java 线程不适合 Goroutine-per-Chain：**

```
场景：监听 10 条链

Java 方案：
10 个 OS 线程 × 1MB 栈 = 10MB 内存（仅栈）
任何线程阻塞等待 RPC 响应 = 占用 1 个 OS 线程
RPC 超时重试 = 线程继续被占用

Goroutine 方案：
10 个 Goroutine × 2KB 栈 = 20KB 内存
阻塞时自动让出 CPU，其他 Goroutine 可用
重试不额外消耗资源
```

### 1.3 GMP 调度模型

```
G (Goroutine)  →  M (Machine/Thread)  →  P (Processor)
    │                    │                    │
    │   轻量级任务      │  OS 线程          │  逻辑处理器
    │   10万+           │  通常=CPU数       │  通常=CPU核心数
    ▼                    ▼                    ▼

┌─────────┐           ┌─────────┐         ┌─────────┐
│   G1    │           │   M1    │  ┌───▶  │   P1    │ Local Queue
│   G2    │  可运行   │   M2    │  │      │ [G,G,G] │
│   G3    │  ───────▶ │   M3    │  │      └─────────┘
│   ...   │           │   ...   │  │           │
│ G10000  │           └─────────┘  │           ▼
└─────────┘                          │      Run in P
     │                               │
     ▼                               │
  全局队列                             │
 Global Queue                          │
    [G,G,G]                           │
                                      │
                              工作窃取 (Work Stealing)
                              P1 任务少 → 从 P2 偷 G
```

**调度优势：**
- **Work Stealing**：P 本地队列空时，从其他 P 窃取任务
- **Handoff**：M 因系统调用阻塞时，P 可转交其他 M
- **CPU 绑定**：P 与 CPU 核心绑定，减少缓存失效

### 1.4 Channel：通信而非共享内存

**传统并发（共享内存）：**
```
两个线程访问共享变量 → 需要加锁 → 容易死锁/竞态
```

**Go 并发（CSP 理论）：**
```
Goroutine A  ──数据──▶  Channel  ──数据──▶  Goroutine B
                  (并发安全)                (独立处理)
```

**Channel 在 Goroutine-per-Chain 中的作用：**
```
各链 Goroutine  ──事件──▶  Channel  ──事件──▶  核心服务
  (独立生产)              (缓冲队列)         (统一消费)
```

## 二、What：Goroutine-per-Chain 是什么

Goroutine-per-Chain 是一种为每条区块链分配独立 Goroutine 的并发监听模式。在多链钱包系统中，每个公链（BTC、ETH、TRX、SOL）都有专属的监听协程，负责该链的区块订阅、交易解析和事件推送。

**核心特征：**
- 一链一协程：每条链对应一个独立的 Goroutine
- 独立 Channel：每个 Goroutine 拥有专属的事件 Channel
- 统一接口：所有链遵循相同的 ChainAdapter 接口
- 汇聚处理：各链事件汇聚到核心服务层进行统一处理

---

## 三、Why：为什么选择 Goroutine-per-Chain

### 3.1 链异构性

不同区块链之间存在本质差异：

| 维度 | BTC | ETH | TRX | SOL |
|------|-----|-----|-----|-----|
| 出块速度 | ~10分钟 | ~12秒 | ~3秒 | ~400ms |
| 交易格式 | UTXO | 账户模型 | 账户模型 | 账户模型 |
| 确认机制 | PoW | PoS | DPoS | PoH+PoS |
| RPC 标准 | JSON-RPC | EIP-1474 | TRON-JSON-RPC | JSON-RPC |

**统一监听的问题：** 单一 Goroutine 无法同时优化处理如此差异的链特性。

### 3.2 故障隔离

**问题场景：**
- 以太坊 RPC 节点故障 → 如果采用统一监听，其他链的监听也会被阻塞
- 某条链出现分叉 → 重试逻辑不应影响其他链的正常监听
- 某条链交易量激增 → 不应挤占其他链的处理资源

**Goroutine-per-Chain 的优势：**
- 单链故障只影响自身
- 独立的错误重试机制
- 各链处理资源互不干扰

### 3.3 扩展性

**新增链支持：**
```
新增适配器 → 实现 ChainAdapter → 启动新 Goroutine → 立即生效
```

无需修改现有链的监听逻辑，符合开闭原则。

### 3.4 性能优化

每条链可以独立优化监听策略：
- BTC：低频轮询（10分钟出块）
- SOL：高频订阅（400ms 出块）
- 按 ChainType 调整连接池大小
- 按出块速度调整 Buffer 大小

## 四、How：如何设计

### 4.1 统一抽象层

所有链适配器实现统一接口：

```
ChainAdapter (interface)
    ├── SubscribeTransactions()   // 订阅链上交易
    ├── GenerateAddress()          // 生成新地址
    ├── BroadcastTx()              // 广播交易
    ├── GetBalance()               // 获取余额
    └── EstimateGas()              // 估算Gas
```

### 4.2 并发模型结构

```
┌────────────────────────────────────────────────────────────┐
│                    钱包核心服务                            │
│                     (Service)                              │
│  ┌──────────────────────────────────────────────────────┐ │
│  │                  Event Channel (汇聚)                │ │
│  └──────────────────────────────────────────────────────┘ │
└────────────────────┬───────────────────────────────────────┘
                     │
     ┌───────────────┼───────────────┬───────────────┐
     ▼               ▼               ▼               ▼
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  BTC    │    │  ETH    │    │  TRX    │    │  SOL    │
│ Goroutine│   │ Goroutine│   │ Goroutine│   │ Goroutine│
└─────────┘    └─────────┘    └─────────┘    └─────────┘
     │               │               │               │
     ▼               ▼               ▼               ▼
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  BTC    │    │  ETH    │    │  TRX    │    │  SOL    │
│ Adapter │    │ Adapter │    │ Adapter │    │ Adapter │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
     │               │               │               │
     ▼               ▼               ▼               ▼
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ Bitcoin │    │Ethereum │    │ Tron    │    │ Solana  │
│  Node   │    │  Node   │    │  Node   │    │  Node   │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
```

### 4.3 事件流转

```
链上事件 → Goroutine 捕获 → Channel 传输 → 核心服务处理
            (独立)          (并发安全)      (汇聚)
```

**Channel 作用：**
- 解耦监听与处理
- 天然并发安全
- 背压控制（处理不过来时自动阻塞）

### 4.4 生命周期管理

**启动：**
```
服务启动 → 初始化所有 Adapter → 为每个链启动 Goroutine → 开始监听
```

**优雅关闭：**
```
关闭信号 → 关闭所有订阅 → 等待 Channel 清空 → 退出 Goroutine → 服务关闭
```

## 五、适用场景

### 适合
- 需要同时监听多个独立数据源
- 数据源之间无强耦合关系
- 对单点故障隔离有要求
- 各数据源处理策略需要独立优化

### 不适合
- 数据源之间存在强事务关系
- 资源极度受限（每个 Goroutine 有内存开销）
- 监听源数量极大（上万级别）

## 六、与替代方案对比

| 方案 | 故障隔离 | 资源利用率 | 复杂度 | 扩展性 |
|------|---------|-----------|-------|-------|
| Goroutine-per-Chain | ✅ 最好 | 中 | 低 | ✅ 最好 |
| 单线程轮询 | ❌ 差 | 高 | 低 | ❌ 差 |
| 线程池 | ⚠️ 中 | ✅ 最好 | 高 | ⚠️ 中 |

---

## 七、总结

Goroutine-per-Chain 是一种为每条区块链分配独立监听协程的并发模式。它通过统一接口抽象链异构性，通过独立 Goroutine 实现故障隔离，通过 Channel 汇聚事件实现统一处理。在多链钱包场景下，这是平衡隔离性、扩展性和复杂度的最优解。
